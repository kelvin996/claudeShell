## 一、提示词基础

### 1.1 什么是提示词（Prompt）

提示词是你与 Claude Code 沟通的自然语言指令。好的提示词能让 Claude 准确理解你的意图，提供高质量的代码和解决方案。

### 1.2 提示词的基本结构

```
[上下文] + [具体任务] + [约束条件] + [期望输出]
```

**示例：**
```
我正在开发一个 React 项目 [上下文]
帮我创建一个用户登录表单组件 [具体任务]
使用 TypeScript 和 Tailwind CSS [约束条件]
包含邮箱、密码字段和表单验证 [期望输出]
```

### 1.3 核心原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **清晰具体** | 明确说明要做什么 | ❌ "写个函数" → ✅ "写一个计算斐波那契数列的函数" |
| **提供上下文** | 说明项目背景 | "使用 Next.js 14 的 App Router" |
| **设定约束** | 指定技术栈和限制 | "不使用第三方库" |
| **分步请求** | 复杂任务拆解 | "先设计数据结构，再实现 API" |

## 二、高效提示词模式

### 2.1 代码生成模式

#### 基础模板
```
请创建一个 [功能名称]，要求：
1. 功能描述：[详细说明]
2. 技术栈：[框架/语言]
3. 输入参数：[参数列表]
4. 返回值：[返回类型和格式]
5. 错误处理：[异常情况]
```

**示例：**
```
请创建一个用户注册函数，要求：
1. 功能描述：验证用户输入并创建新用户账户
2. 技术栈：TypeScript + Node.js + Express
3. 输入参数：email（邮箱）、password（密码）、name（用户名）
4. 返回值：Promise<User> 或抛出错误
5. 错误处理：邮箱格式无效、密码强度不足、邮箱已注册
```

#### 组件开发模板
```
创建一个 [组件名称] 组件：

功能需求：
- [需求1]
- [需求2]

技术要求：
- 框架：React/Vue/Svelte
- 样式：Tailwind CSS / CSS Modules
- 类型：TypeScript

交互行为：
- [交互说明]

参考设计：
- [设计稿链接或描述]
```

## 四、高级技巧

### 4.1 多轮对话技巧

#### 渐进式细化
```
第一轮：概述需求
"我想实现一个文件上传功能"

第二轮：补充细节
"需要支持图片和PDF，最大10MB"

第三轮：添加约束
"使用 React Dropzone，显示上传进度"

第四轮：优化完善
"添加拖拽效果和文件预览"
```

#### 角色扮演
```
请扮演一个资深后端架构师，帮我设计一个高并发的订单系统。

要求：
- 日活用户 100 万
- QPS 峰值 10000
- 数据一致性要求高
- 支持分布式部署
```

### 4.2 上下文管理

#### 项目上下文注入
```
项目信息：
- 名称：电商后台管理系统
- 技术栈：Next.js 14 + TypeScript + Prisma + PostgreSQL
- 架构：App Router + Server Actions

编码规范：
- 使用函数式组件
- 使用 Tailwind CSS
- 优先使用 Server Components

当前任务：[具体任务]
```

#### 文件上下文引用
```
参考项目中的以下文件：
- src/lib/db.ts - 数据库连接
- src/types/user.ts - 用户类型定义
- src/middleware/auth.ts - 认证中间件

基于以上文件，实现 [功能]
```

### 4.3 思维链提示

```
请一步步思考并实现用户权限系统：

1. 先分析需求，列出需要的功能点
2. 设计数据模型和关系
3. 规划 API 接口
4. 实现核心逻辑
5. 添加测试用例

每一步请先说明思路，再编写代码。
```

### 4.4 约束与限制

```
实现 [功能]，请遵守以下约束：

必须：
- 使用 TypeScript strict 模式
- 所有函数必须有 JSDoc 注释
- 错误处理使用 Result 模式

禁止：
- 不使用 any 类型
- 不使用 var 声明
- 不忽略 TypeScript 错误

性能要求：
- 初始加载 < 3s
- 交互响应 < 100ms
```

## 六、常见错误与改进

### 6.1 模糊请求

| ❌ 错误示例 | ✅ 改进示例 |
|------------|------------|
| "写个函数" | "写一个 TypeScript 函数，接收字符串数组，返回去重后的数组" |

## 八、最佳实践

### 8.1 提示词编写检查清单

- [ ] 明确说明了任务目标
- [ ] 提供了必要的上下文
- [ ] 指定了技术栈和约束
- [ ] 描述了期望的输出格式
- [ ] 提供了相关代码或文件引用
- [ ] 说明了验收标准

### 8.2 迭代优化策略

```
第一版提示词 → 获得输出 → 评估质量
     ↓
"请修改 X 部分，改为 Y 方式"
     ↓
"添加 Z 功能，优化性能"
     ↓
"生成单元测试"
     ↓
最终版本
```

### 8.3 高效沟通技巧

1. **善用代码块**：用 ``` 标记代码，便于 Claude 识别
2. **结构化表达**：使用标题、列表、表格组织信息
3. **引用文件**：使用 `@文件名` 或粘贴关键代码
4. **明确优先级**：用"必须"、"应该"、"可选"区分需求

## 九、进阶技巧

### 9.1 自定义指令

在项目中创建 `CLAUDE.md` 文件，定义项目特定的指令：

```markdown
# 项目指令

## 技术栈
- Next.js 14 App Router
- TypeScript strict mode
- Tailwind CSS
- Prisma ORM

## 编码规范
- 使用函数式组件
- 优先使用 Server Components
- 使用 Zod 进行数据验证

## 命名约定
- 组件：PascalCase
- 函数：camelCase
- 常量：UPPER_SNAKE_CASE

## 禁止事项
- 不使用 any 类型
- 不使用 var 声明
- 不在客户端组件中使用服务端代码
```

### 9.2 多文件协作

```
我需要修改用户认证相关的功能，涉及以下文件：

1. src/app/api/auth/login/route.ts - 登录 API
2. src/lib/auth.ts - 认证逻辑
3. src/components/LoginForm.tsx - 登录表单

请分析这些文件的关联关系，然后实现 [具体修改]
```

### 9.3 渐进式开发

```
让我们分步骤开发用户管理模块：

第一步：设计数据库模型（等确认后再继续）
第二步：实现 API 接口
第三步：创建前端页面
第四步：添加权限控制
第五步：编写测试用例
